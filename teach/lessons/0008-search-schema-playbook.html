The win: You leave with the exact set of site-owner levers that verifiably move SharePoint retrieval quality — crawled→managed mapping, the three attribute dials, and the knowledge-source search filters — and you know which lever to reach for when the wrong document keeps winning a top-3 slot in your agent's answers.

The stakes

Top-3 grounding makes precision decisive

From lesson 07: only the top 3 search results ground each generative answer (sharepoint-no-response). That single mechanic is why this playbook exists. Every lever below serves one goal: get the right document into those three slots and keep the wrong ones out.

Directional

Point the knowledge source at the narrowest URL that holds authoritative content, archive superseded near-duplicates, and keep one canonical document per topic. This is inference from the verified top-3 mechanic — not a quoted Microsoft instruction — but it follows directly.

The pipeline

Crawled → managed: nothing is searchable until it's mapped

SharePoint search discovers metadata as crawled properties, but only managed properties participate in queries and results (crawled-and-managed-properties-overview). New metadata joins the index only after you map it and the content is re-crawled.

Verified

"If you add a managed property, you must map it to a crawled property to get content into the index. After the site, library, or list is crawled, users can search for the content and metadata of new or changed managed properties… [changes] take effect only after the content is re-crawled." — manage-search-schema

Field note

The working sequence: SharePoint admin center → Search → Manage Search Schema; find or create the managed property; map it to the crawled property; await a re-crawl; then verify the content comes back in plain SharePoint search before expecting the agent to use it.

Directional

The crawled/managed mechanics are documented on SharePoint Server pages. In SharePoint Online the schema is managed per-tenant with less crawl-scheduling control — you may need to re-save items or simply wait. Treat attribute semantics as authoritative and crawl-timing as directional.

The dials

Searchable, Queryable, Retrievable

Each managed property carries independent attributes — the dials a site owner turns (search-schema-overview).

AttributeWhat it doesRAG use
SearchableAdds content to the full-text index; matched by free-text queriesON for content the agent should find by keyword or semantics
QueryableEnables property-scoped queries (e.g. author:Smith) when the property is namedTargeted filtering — within Copilot Studio's own query limits
RetrievableAllows the value to be returned in resultsDisable to hide noisy properties from grounding
Refinable / SortableEnable refiners and sortingNot supported in the modern search experience — skip
Verified

"If you set a managed property to be searchable, the content is added to the index… users can query for author:Smith… to prevent the content in a managed property from showing up as search results, you can disable the retrievable setting." — manage-search-schema

Gotcha

Refinable and Sortable are both "not supported in the modern search experience" (search-schema-overview). Time spent configuring refiners for this pipeline is time wasted — the three dials that matter are Searchable, Queryable, and Retrievable.

The filters

Search query filters: the highest-ROI knob

The one lever that needs no admin center at all: on the SharePoint knowledge source itself, Copilot Studio lets you set search query filter parameters on Title, Author, Modified by, and Modified on (knowledge-add-sharepoint).

Verified

"To improve the performance of your agent's SharePoint knowledge source, specify search query parameters… For example, you could specify that you only want searches for items that were modified in the last six months." Build your filters "to include or exclude information." — knowledge-add-sharepoint

Field note

With only three grounding slots, every filter that shrinks the candidate set toward authoritative content raises the odds the right doc lands in the top 3. Concretely: filter Modified on to the last N months to exclude stale docs, scope Author to the owners of authoritative content, and use Title patterns to include or exclude whole document families.

The boundary

What schema tuning cannot buy

Schema and filters improve what is findable and returned — precision and disambiguation of grounded answers. They do not repeal the structural limits from lesson 07.

Gotcha

Making a column Queryable does not make the agent answer "list / count / filter by column value" queries — Copilot Studio explicitly doesn't support them at the knowledge-source level (requirements-quotas). File-by-name questions stay dead too. Schema tuning sharpens retrieval; it never adds database-style query capability.

Field check
Field check

Answer from memory before clicking — retrieval is what makes it stick.

1. You add a new managed property and map a crawled property to it. When does the content become searchable?

Schema changes take effect only after the content is re-crawled. In SharePoint Online you have limited crawl control — re-save items or wait, then verify in plain search first.

2. Which attribute do you disable to keep an indexed property's value out of search results?

Disabling Retrievable prevents a managed property's content from showing up in results — the documented way to hide noisy or internal properties even when they're indexed.

3. Why do the Copilot Studio search query filters (e.g., Modified on) matter so much for answer quality?

Filters shrink the candidate set toward authoritative content, which directly improves the odds the right document wins one of only three grounding slots. They change nothing about ceilings or unsupported query types.

4. You make a Status column Queryable in the schema. Can the agent now answer "List all files where Status = Approved"?

Copilot Studio doesn't support column-value filtering at the knowledge-source level no matter what the schema allows. Structured queries belong in Dataverse, a flow, or a custom tool — not SharePoint knowledge.

5. Which filter fields does the SharePoint knowledge source actually support?

The documented filter fields are Title, Author, Modified by, and Modified on. Folder and column-value filtering are not among them — and lesson 07 explains why they never will be at this layer.
