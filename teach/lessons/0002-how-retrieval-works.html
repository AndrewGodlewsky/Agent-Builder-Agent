How retrieval actually works

ScopeThe orchestration-driven RAG pipeline: who triggers search, the four steps, and the gates
Builds on01 · Knowledge vs. Instructions

The win: After this lesson you can trace any question a user asks your agent through the retrieval pipeline — predict whether knowledge fires, how many chunks come back, and where your configuration actually has leverage.

Who decides

The orchestrator triggers retrieval — per turn, automatically

You never wire "call knowledge here" on the default path. Under enhanced (generative) orchestration — the current default — the runtime evaluates each turn and decides whether grounding is needed and which sources to search.

Verified

"The enhanced orchestration runtime automatically determines which knowledge sources to search based on the user's question." (knowledge-copilot-studio, agents-experience)

This is why lesson 01's boundary matters: retrieval is the orchestrator's call, made from the user's question and your sources' names and descriptions — not from anything your instructions say.

The pipeline

Four steps, every grounded turn

StepWhat happensYour leverage
1 Query rewriting. The raw message becomes an optimized search query, enriched with conversation context (prior turns, location, time). None — automatic
2 Content retrieval. The rewritten query runs against all configured sources; Copilot Studio pulls the top three results from each source. Structural only — which sources exist and where
3 Summarization. Retrieved chunks become a grounded answer. Custom instructions apply here: tone, formatting, safety, brevity. Instructions bite here
4 Safety validation. Grounding, provenance, and relevance checks; the answer ships with citations. None — automatic
Verified

"After rewriting the query, the system runs it against all the knowledge sources you set up. Copilot Studio gets the top three results from each source." (retrieval-augmented-generation)

Gotcha

Top-3-per-source means adding sources widens coverage but never deepens it — each source contributes at most ~3 chunks no matter how relevant its content is. This is why RAG handles factual snippet questions well and whole-document reasoning badly, and why an agent can look well-grounded while missing most of a long document.

For public-website sources, retrieval runs through Bing Custom Search, restricted to your configured domains, with the same separation of retrieval from summarization plus grounding, provenance, and semantic-similarity cross-checks.

Indexing

How your files become retrievable

Uploaded unstructured files are chunked and vector-indexed (semantic embeddings) and stored in Dataverse. At query time, Copilot Studio retrieves the chunks most relevant to the rewritten query and grounds the response on them.

Directional

Chunk size is not configurable and not published by Microsoft (per practitioner reporting). Plan on it: you cannot tune chunking — your levers are document structure and source selection, not index parameters.

A doc nuance

Fallback vs. proactive search — both framings appear

You'll meet two descriptions of when knowledge search fires, and both are correct for their context:

  • Classic / SharePoint generative-answers wording: knowledge search runs as a fallback — it triggers when no topic answers the user, and for SharePoint it searches the configured URL plus all subpaths. (knowledge-add-sharepoint)
  • Modern generative orchestration (the default): the orchestrator invokes knowledge search proactively via the Universal Search Tool — not strictly as a last resort.
Field note

On generative orchestration, think "the orchestrator searches knowledge whenever it judges the turn needs it," not "knowledge only fires when topics miss." The fallback framing is incomplete for the modern default — not wrong for classic. Don't design your agent around last-resort retrieval.

The gates

Grounding enforcement and source scoping

Two settings decide what retrieval is allowed to do for your agent:

1. Allow ungrounded responses. This gate decides whether the agent may ever answer without grounding.

Verified

"When you turn off this setting, the agent blocks any response generated in a turn where it didn't use a knowledge source or tool… the fallback topic triggers." (knowledge-copilot-studio)

Off means every turn must ground on a knowledge source or tool, or the agent refuses and routes to the fallback topic — the setting for "answer only from approved content" agents. On means the agent may fall back to general model knowledge — more flexible, less contained.

2. Source scoping. By default, every source on the agent's Knowledge page is searched. To restrict a scenario to specific sources, use a generative answers node in a topic: node-level sources override the agent-level list, which then acts as a fallback. As lesson 01 established, this structural scoping is the only supported way to steer retrieval — instructions cannot.

Field note

Separate surface, don't conflate: the M365 Copilot Retrieval API is a developer API over the M365 hybrid index (KQL-scopable), distinct from Copilot Studio's point-and-click knowledge feature described here.

Field check
Field check

Answer from memory before clicking — retrieval is what makes it stick.

1. Under generative orchestration (the default), when does your agent search its knowledge sources?

The orchestrator decides per turn, proactively, via the Universal Search Tool. "Only when topics miss" is the classic fallback framing — incomplete for the modern default.

2. How much content does retrieval return per query?

Top three per source is the defining constraint: more sources widen coverage but never deepen any single source's contribution.

3. At which pipeline step do your custom instructions actually apply?

Instructions bite only at summarization (tone, formatting, safety, brevity). Query rewriting and retrieval are automatic and untouchable — lesson 01's boundary, seen from inside the pipeline.

4. You turn "Allow ungrounded responses" off. What happens on a turn where the agent used no knowledge source or tool?

With the setting off, any response generated without grounding on a knowledge source or tool is blocked and the fallback topic triggers — the high-assurance, approved-content-only configuration.

5. What is the supported way to restrict a scenario to specific knowledge sources?

Node-level sources override the agent-level list (which becomes fallback). Instructions can't steer retrieval, and chunking isn't configurable — structure is the only lever.
