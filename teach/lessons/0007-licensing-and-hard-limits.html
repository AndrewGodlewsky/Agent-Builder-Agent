The license

What M365 Copilot actually unlocks

Your tenant has a Microsoft 365 Copilot license in the same tenant as the agent. That is the gate for two things: tenant graph grounding with semantic search — retrieval rides the tenant semantic index instead of plain keyword search — and a much higher file-size ceiling for SharePoint generative answers.

Verified

"For better search results from SharePoint and support of files up to 200 MB in size, use a Microsoft 365 Copilot license in the same tenant… and turn on tenant graph grounding with semantic search." — requirements-quotas

Directional

Microsoft describes the trade as "a small increase in latency" for "greater volume of context, with greater precision" (knowledge-add-sharepoint). The precision claim is Microsoft's self-assessment — no independent benchmark exists. Turn it on anyway; the latency cost is the only known downside.

The switches

Two toggles convert the license into capability

The license alone changes nothing. Two settings have to be ON: Enhanced search results and tenant graph grounding with semantic search (knowledge-add-sharepoint).

Verified

"If you have a Microsoft 365 Copilot license… the maximum file size is 200 MB. You must also turn on the Enhanced search results feature." — sharepoint-no-response

Gotcha

With Enhanced search results left OFF — the no-license default — you are capped at 7 MB even though you're licensed. If large documents are mysteriously ignored, check the toggle before anything else.

The silent over-ceiling failure

Verified

"Larger files can be stored in SharePoint and are returned by a Microsoft Graph search, but aren't processed by generative answers. As an alternative, you can upload your own files, which can be up to 512 MB in size." — sharepoint-no-response

Gotcha

An over-ceiling file appears as a search hit but contributes nothing to the answer. No error, no warning — it looks like the agent "found" the document. This is worse than a hard failure because it hides in testing. Audit your library for files over the ceiling; for the big ones, use the 512 MB file-upload path instead.

The impossible

Four queries no setting can fix

Some question shapes are structurally unsupported by a SharePoint knowledge source. No prompt, instruction, or schema change enables them — this is a property of how the connector retrieves, not a tuning gap.

Verified

"Queries to a SharePoint knowledge source that reference a file or document name can't be answered." Folder-level metadata filtering "isn't supported" — the agent can't list or count files in a folder — and "queries related to column metadata filtering properties aren't supported." — requirements-quotas

Cannot be answeredExample
Reference a file by name"What's in policy-v3.pdf?"
List files in a folder"What documents are in the HR folder?"
Count files"How many files are in this library?"
Filter by column value"List all files where Status = Approved"
Directional

Instructions can't steer retrieval either: Microsoft's guidance is to remove any instruction that tries to influence document retrieval. Instructions govern summarization after retrieval — so you cannot instruction your way around any of the four shapes above.

Field note

Design around, not against. If your users genuinely need "list / count / filter" answers, that workload belongs in a different mechanism — a Dataverse table, a Power Automate flow, or a custom tool — not in SharePoint knowledge. Decide this at design time, before the first bug report teaches you.

Unconfirmed

Open tension: the SharePoint search schema can make column metadata queryable, yet Copilot Studio still refuses column-value filtering at the knowledge-source level. Whether managed metadata helps grounding disambiguation in any other way is unresolved — test against your tenant before betting on it.

Field check
Field check

Answer from memory before clicking — retrieval is what makes it stick.

1. Your tenant has an M365 Copilot license, but Enhanced search results is OFF. What's the SharePoint ceiling for generative answers?

The license is necessary but not sufficient — Enhanced search results must be ON, or you stay at the 7 MB no-license default despite being licensed.

2. A 300 MB PDF sits in the connected SharePoint library. What happens when a user's question matches it?

Over-ceiling files are returned by Graph search but aren't processed by generative answers — a silent failure that looks like a successful retrieval.

3. Users keep asking "How many contracts are in the Legal folder?" Can any instruction or schema change ever make the SharePoint knowledge source answer this?

Folder listing and counting are structurally impossible for a SharePoint knowledge source. Instructions govern summarization after retrieval, and schema tuning can't add this capability — route the workload to Dataverse, a flow, or a custom tool.

4. How many search results ground each generative answer from the SharePoint connector?

Only the top 3 results ground each answer — which is why retrieval precision and deduplication matter so much (lesson 08 works those levers).

5. A user asks "What's the mitigation in policy-v3.pdf?" What's the correct design response?

File-by-name queries can't be answered by a SharePoint knowledge source, and instructions can't steer retrieval. The fix is structural: handle that query shape elsewhere, or teach users to ask by topic instead.
