Permission trimming & security

ScopeHow SharePoint knowledge enforces permissions at query time — and why "no response" is usually a permissions problem
Builds on05 · The two SharePoint RAG paths
Sourceresearch/sharepoint-rag-optimization/02

The win: When a user of your agent reports "it just says nothing," you diagnose permissions first — not content, not config — and you can explain exactly why the agent stayed silent. You also know to audit your site's permissions before rollout, because the agent will surface everything users can technically read.

The mechanism

Trimming happens at retrieval time, per user

Across every SharePoint path — connector, file upload, and the Retrieval API — permission trimming happens live, at query time, using the querying user's Microsoft Entra ID (delegated permissions). The system checks access against the source before returning or summarizing anything, and the user needs at least Read permission on the relevant sites and files (knowledge-add-sharepoint).

This holds even for Path B, where content was copied into Dataverse: no permission information is stored locally — every check goes back to SharePoint.

Verified

"Even though the system stores chunks and indexes locally in Dataverse, it performs a live check on the queries to make sure the current user has access to the data before providing a summary or response… The system doesn't store content permission information locally… all permission checks live with the source." — knowledge-unstructured-data

Field note

Governance upside: you cannot use a SharePoint knowledge source to broadcast restricted content to users who lack access. The trimming is identity-based and enforced at the source, every query — the same model the Retrieval API documents ("individuals can only get results from the content they are allowed to access").

The #1 troubleshooting trap

The silent "no response"

When the user lacks access, the agent does not error. It returns nothing — silently.

Verified

"Generative answers over SharePoint rely on delegated permissions… a user must have read permissions on the relevant sites and files, or no search results will be returned… nor any errors or exceptions." — sharepoint-no-response

Gotcha

There is no error, no exception, no "access denied" — just "no response." Nothing in the transcript tells you the cause was permissions. If you treat this as a content or configuration bug, you will burn hours tuning documents that were never the problem.

Now combine this with lesson 05's top-3 rule: if the three top-ranked results are documents the user can't see, they are trimmed away and nothing remains to ground on — the user gets no answer even though relevant, accessible content exists at rank 4 and below.

Verified

Inaccessible top hits can leave a user with nothing: permission trimming (verified) plus top-3 grounding (verified) means a user who can only see documents ranked 4th or lower gets no answer at all. This is an inference from two verified facts, flagged as such in the research (sharepoint-no-response).

Testing implication

Test as a real end user — never as yourself

Trimming is per-identity, and you are the site owner. An admin or site-owner tester sees results a regular user won't. Always test with an account that has the actual permissions of your target audience (knowledge-add-sharepoint).

Directional

If answers work for you but users report "no response," suspect permissions first — before content or configuration. This triage order comes from the verified mechanics but isn't itself a documented procedure.

Field note

For your agent: keep (or borrow) a test account with exactly the target audience's permissions, and make a permission-trimmed test pass part of every rollout. Your own account is the least representative one in the tenant.

The flip side

Oversharing — trimming is only as good as your permissions

Permission trimming enforces what users can access, not what they should access. If a library is over-permissioned (say, "Everyone" has Read), the agent will happily ground answers on it for everyone who asks.

Directional

Copilot agents surface content users technically can access but perhaps shouldn't — latent oversharing becomes suddenly very visible and very queryable. Site owners should audit and tighten permissions before rollout. (Practitioner-sourced; Microsoft's privacy doc confirms the trimming model, but the remediation guidance is community wisdom.)

One more enforcement layer, verified for the connector path: it respects sensitivity labels in addition to Read permissions (knowledge-unstructured-data).

Field check
Field check

Answer from memory before clicking — retrieval is what makes it stick.

1. When does permission trimming happen for SharePoint knowledge sources?

Trimming is a live check at retrieval time, every query — even for Path B, where content sits in Dataverse. No permission information is stored locally; all checks live with the source.

2. A user without Read permission asks a grounded question. What happens?

No search results are returned, "nor any errors or exceptions" — the agent just gives no response. This silence is the #1 troubleshooting trap.

3. Which identity does the permission check use?

Trimming is delegated: it uses the identity of the person asking, not the maker's or a service account's. That's why results differ per user — and why your admin account is a misleading tester.

4. Answers work for you, but users report nothing. First suspect?

You're the site owner — you see everything. When it works for you and fails for them, the per-identity trimming is the prime suspect, before content or configuration.

5. A user can only access documents ranked fourth or lower. Result?

Only the top 3 results ground an answer. If trimming removes all three, nothing remains to ground on — the user gets no response even though relevant, accessible content exists further down.
