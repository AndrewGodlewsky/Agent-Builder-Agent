The win: You can look at any piece of document-prep advice for your SharePoint-grounded agent and instantly tag it — proven, plausible, or test-it-first — then spend your effort where the evidence actually is instead of polishing headings a spec never promised to reward.

The real skill

Most doc-authoring advice here is unproven — that's the lesson

The research hunted hard for primary-sourced guidance on authoring documents for better chunks: OCR of scanned PDFs, heading structure, alt text, title hygiene. Microsoft largely does not document it. So this lesson's first job is calibration: every claim below wears one of three stamps, and the stamp tells you what to do with it.

Field note

Unconfirmed does not mean wrong. It means: if your retrieval quality depends on it, prove it in your tenant with a test question only that document can answer. The unconfirmed items in this lesson are a test list, not a spec.

Type 1

Large PDFs

Two things are solid here, and both are about size (see Copilot Studio quotas and limits).

Verified

File-size ceilings apply: 512 MB for PDFs via the licensed connector or file upload; general SharePoint files cap at 200 MB (with tenant graph grounding and semantic search on).

Verified

Oversized files fail silently: Graph search still returns them as hits, but generative answers never process them (SharePoint no-response troubleshooting).

Gotcha

Because the oversized file still appears as a search hit, nothing looks broken — the agent just quietly answers without it. If a big PDF is your authoritative source, this failure mode is invisible until you test for it.

So the one verified-backed authoring action for huge PDFs: split them into smaller, topic-scoped PDFs. Each piece stays under the ceiling and retrieves more precisely, which helps its odds of landing in the top-3 results that ground each answer.

Unconfirmed

OCR for scanned, image-only PDFs. RAG needs text, and no primary source confirms whether SharePoint's crawl or Copilot Studio's ingestion OCRs image-only PDFs. Safest assumption: OCR them yourself before uploading, then test a scanned PDF end-to-end before trusting it.

Type 2

Office docs — Word, PowerPoint, Excel

Verified: .docx and .pptx get the same 512 MB ceiling under your license. Beyond size, the strongest evidence comes from a neighboring surface — the Retrieval API (lesson 10) — and it transfers only partially.

Verified

For the Retrieval API: table text is retrievable only from .doc/.docx/.pptx, and images and charts are not retrievable at all. Whether Copilot Studio's built-in connector behaves identically is unconfirmed — treat these limits as a worst-case planning assumption, not a settled fact about the connector.

Gotcha

An answer that lives only inside a PowerPoint diagram or an embedded image may never ground a response — and you'll get a confident answer built from whatever text was retrievable. Put the answer in text too: a caption, slide notes, or a paragraph.

Directional

Keep the document Title property meaningful (not just the filename). It's verified that Title is a filter field on the SharePoint knowledge source (Add SharePoint knowledge), so descriptive titles are low-risk, high-plausibility for retrieval and disambiguation.

Unconfirmed

Real headings and clear structure in Word docs. Standard RAG practice says structure aids chunk boundaries, but Microsoft doesn't confirm it affects Copilot Studio chunking. Do it anyway — it's free and helps humans — just don't bank retrieval quality on it.

Field note

Excel is weak for narrative RAG: spreadsheets are tabular, and retrieval is scoped to factual Q&A over text. For structured lookups, use a Dataverse table knowledge source, not an Excel file in SharePoint.

Type 3

Native SharePoint pages and lists

Pages are the best-behaved content type you have; lists are the most misunderstood.

Verified

The tenant semantic index is built from text-based SharePoint Online files (Semantic index for Copilot) — clean, native page text is exactly what it's built for. Use pages for authoritative narrative content: policies, how-tos.

Verified

Lists are not a query surface. Even if you map list columns to managed properties (lesson 08), column-value filtering queries stay unsupported at the knowledge-source level — "list all items where X" cannot be answered and cannot be fixed with instructions. Use Dataverse.

Directional

Keep page content focused — one topic per page. Narrow pages improve the odds that the right page, not a sprawling omnibus one, lands in the top 3.

Cross-cutting

The top-3 discipline

Every prep decision serves one goal: make the single authoritative answer land in the top 3 search results, because only those ground each generative answer (troubleshooting doc). That makes curation a verified lever, not a nicety:

  • Deduplicate — near-identical docs split relevance and crowd the top 3. Verified
  • One canonical doc per topic; filter out stale versions with search query filters (lesson 08). Verified
  • Right-size documents — omnibus docs may retrieve a chunk that misses the point; topic-scoped docs plausibly retrieve more precisely. Test it. Unconfirmed
Field note

Microsoft's closest thing to an authoring playbook is three directional pages worth reading against your own tenant: Optimize SharePoint content for Copilot, Optimize content for the Retrieval API, and Generative answers with SharePoint/OneDrive. They may confirm some unconfirmed items for your setup.

1. Tier recall: which of these document-prep claims is actually verified by primary Microsoft documentation?

The silent oversized-file behavior is verified in the SharePoint no-response troubleshooting doc. OCR and heading structure are both unconfirmed — sensible to do, but nothing primary-sourced promises they help this pipeline.

2. A PDF in your connected library is over the size ceiling. How does the failure show up?

Verified: oversized files are still returned by Graph search but are not processed by generative answers. Nothing errors and nothing disappears — the file just never contributes, which is why you must test for it.

3. What does the research actually say about OCR for scanned, image-only PDFs?

No primary source confirms OCR behavior on either path — it's an open question. The safe play: make scanned PDFs text-searchable yourself, then verify with a question only that PDF can answer.

4. Why does deduplicating near-identical documents matter so much for answer quality?

Only the top 3 search results ground each answer (verified). Near-duplicates compete with each other for those slots and can push the one authoritative document out.

5. You need the agent to answer "list every item where Status is Open." What's the right move?

Verified: column-value filtering queries are unsupported at the knowledge-source level even when columns are mapped to managed properties. Lists aren't a database query surface for the agent — Dataverse is.
