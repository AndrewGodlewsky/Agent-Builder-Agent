The third path

What the Retrieval API actually is

You already know two SharePoint RAG paths (lesson 05): the point-and-click connector and Dataverse file upload. The Microsoft 365 Copilot Retrieval API is a third, code-level grounding option — distinct from both.

Verified

The Retrieval API "returns relevant text chunks from the hybrid index that powers Microsoft 365 Copilot… without the need to replicate, index, chunk, and secure your data in a separate index." It grounds from SharePoint, OneDrive, and Copilot connectors, and is scopable with Keyword Query Language (KQL) — URLs, date ranges, file types.

Field note

Translation: it's a query door straight into the tenant semantic index your M365 Copilot license already maintains. No sync cadence, no copied files, no second index to secure. But it's an API — you write the code that calls it — and it's currently in Preview.

The constraint

Format limits — the sharpest fact in this lesson

The API's documentation is unusually explicit about what it can and cannot retrieve, and these limits are the main thing to memorize.

Verified

"Semantic retrieval and hybrid retrieval is only supported for .doc, .docx, .pptx, .pdf, .aspx, and .one file extensions. All other file extensions only support lexical retrieval… Retrieval from text in tables is limited to .doc, .docx, and .pptx… Retrieval from nontextual content, including images and charts, is not supported." (Retrieval API overview)

Gotcha

These limits are documented for the Retrieval API specifically. Whether Copilot Studio's built-in SharePoint connector has identical format handling is unconfirmed — an open question in the research. Don't assume parity, but since both ride the same underlying index, treat these limits as a reasonable worst-case planning assumption: if an answer lives only in an image or a PDF table, don't bet on any path retrieving it until you've tested.

The decision

Which grounding path, when

Use…When
Built-in SharePoint knowledge source (connector) Default for Copilot Studio agents. Point-and-click, real-time, permission-trimmed, 25 URLs, top-3 grounding. Best when the maker experience and native integration matter. (Add SharePoint knowledge)
File upload (Dataverse) Small, curated, stable document sets; need the 512 MB ceiling; OK with 4–6 h sync; want content isolated in Dataverse. (Knowledge in Copilot Studio)
Retrieval API Custom, code-built grounding — a pro-code agent or app that queries the M365 hybrid index directly with KQL scoping and no separate index to maintain. More control, more build effort, Preview status.
Directional

For most site-owner-driven Copilot Studio scenarios — including yours — the built-in connector is the right default, and the search-schema/filters playbook (lesson 08) is where your effort pays off. Reach for the Retrieval API only when you're building a custom grounding pipeline and need KQL-level control.

The common thread

All three paths ride your tenant semantic index

Whether grounding flows through the connector, file upload, or the Retrieval API, licensed retrieval rides the same automatically built tenant semantic index.

Field note

This is why the document-hygiene work from lesson 09 and the permission work are never wasted: improving your SharePoint content and permissions improves every grounding surface at once. Pick the path per scenario; the content investment transfers.

Field check
Field check

Answer from memory before clicking — retrieval is what makes it stick.

1. How does the Retrieval API ground answers without you building a new index?

Verified: the API returns text chunks from the hybrid index that powers M365 Copilot, "without the need to replicate, index, chunk, and secure your data in a separate index." Copying into Dataverse is the file-upload path, not this one.

2. Which content is explicitly not retrievable via the Retrieval API?

Verified: "Retrieval from nontextual content, including images and charts, is not supported." Table text in .docx works (Office formats only), and .aspx is on the semantic-retrieval list.

3. Table text is semantically retrievable from which file formats only?

Verified: "Retrieval from text in tables is limited to .doc, .docx, and .pptx." Notably that excludes PDFs — a table inside a PDF is not promised to be retrievable.

4. When should you reach for the Retrieval API instead of the built-in connector?

The API is for custom, pro-code grounding pipelines that want KQL-level control over the M365 hybrid index. The connector is already real-time — that's never a reason to leave it.

5. Do the Retrieval API's format limits also apply to the built-in connector?

Connector parity is an open question — no primary source confirms it either way. Since both paths share the tenant index, plan conservatively: assume the limits hold until an end-to-end test in your tenant proves otherwise.
