Chart A

Instructions, knowledge, or topic?

Where a piece of content belongs. The one rule: knowledge = facts the agent retrieves and cites; instructions = behavior, routing, tone, and any examples. (research: builder recommendations)

Facts the agent should look up and cite — docs, policies, site content?
YES →
Knowledge source. That's exactly what RAG is for — factual Q&A over your content. → Lesson 01 · Knowledge vs. instructions
NO ↓
Behavior — persona, tone, routing, formatting, or example/template responses?
YES →
Instructions. Examples go here as few-shot input→output pairs — never as knowledge docs (top-3 retrieval may skip them). → Lesson 01 · Knowledge vs. instructions
NO ↓
A deterministic flow — or a turn that must draw only from specific sources?
YES →
Topic with a generative answers node — node-level sources override the agent-level list. → Lesson 04 · Builder playbook
NO ↓
Trying to steer retrieval? ("Prefer doc X", "search only folder Y") — unsupported. Microsoft says remove it; instructions can't modify retrieval. Scope structurally instead. → Lesson 01 · Knowledge vs. instructions

Chart C

Which SharePoint path — connector or upload? (and when the Retrieval API)

The two SharePoint RAG paths are architecturally different: the connector queries GraphSearch live; file upload copies into Dataverse on a 4–6 h sync. The Retrieval API is a third, code-level option. (research: how SharePoint RAG works · research: Retrieval API vs. built-in)

Need file-by-name lookups, folder listing/counting, or column-value filtering?
YES →
Stop — no SharePoint path supports these. Structurally impossible; instructions can't fix it. Use a Dataverse table, a flow, or a custom tool. → Lesson 05 · Two SharePoint RAG paths
NO ↓
Building a custom pro-code app or pipeline that needs KQL-scoped queries over the M365 hybrid index?
YES →
M365 Copilot Retrieval API (Preview). Keeps data in place, no separate index; semantic retrieval for Office/PDF only, no images/charts. → Lesson 10 · Retrieval API vs. built-in
NO ↓
Living library — needs real-time freshness, per-user permission trimming, sensitivity labels?
YES →
Path A — SharePoint connector (GraphSearch, real time). Turn ON Enhanced search results + tenant graph grounding; set query filters; mind the 7/200 MB gate. → Lesson 05 · Two SharePoint RAG paths
NO ↓
Curated, stable set — files over the connector ceiling, or upload-only formats (.md/.txt)? OK with a 4–6 h sync lag?
YES →
Path B — file upload (Dataverse, 512 MB/file). SharePoint-synced files keep live permission checks; local uploads are visible to every agent user. → Lesson 05 · Two SharePoint RAG paths
NO ↓
Default to Path A — the connector. Spend your effort on the site-owner playbook: query filters, narrow URLs, dedupe, search schema. → Lesson 05 · Two SharePoint RAG paths
Constant on every path

Only the top 3 search results per source ground each answer — whichever box you land in, precision and dedupe beat volume. And all licensed paths ride the same tenant semantic index, so content and permission hygiene are never wasted. (research 01 §4 · research 06 §4)
