Knowledge vs. Instructions: the architectural line

ScopeWhat instructions do, what knowledge does, and why neither can do the other's job
Builds onNone
Sourceresearch/copilot-studio-knowledge/01–02

The win: After this lesson you can look at any piece of content destined for your agent — a tone rule, a policy document, a response example — and place it in instructions or knowledge instantly, and defend the call.

The core distinction

Instructions drive behavior. Knowledge supplies facts.

Copilot Studio gives you two configuration surfaces that look like they might overlap. They don't. Per Microsoft's instructions authoring guidance, an agent depends on instructions to decide which resources (tools, knowledge, topics, other agents) to call, to fill inputs for any tool it invokes, and to generate the response. Instructions are the agent's operating manual.

Knowledge is grounding data — the reference library. The knowledge overview puts it plainly: without knowledge, the agent relies only on its general training and instructions. Knowledge supplies content that answers are built from; it decides nothing.

Verified

Instructions are the behavior mechanism — they decide what resources to call, fill tool inputs, and shape response generation. Knowledge is grounding data retrieved via RAG. The split is architectural, not stylistic. (authoring-instructions, knowledge-copilot-studio)

One dependency runs between the surfaces: instructions can reference a knowledge source by name (the / menu in the instructions editor surfaces attached sources), but the docs are explicit that an agent can't act on instructions to use tools or knowledge sources it doesn't have. You cannot smuggle knowledge into behavior, and you cannot point at a source you never attached.

Two jobs knowledge cannot do

Not instructions — and not a few-shot example set either

The first misuse is obvious once you see the split: a document in knowledge cannot define behavior, persona, or flow. It is data that may or may not be retrieved on a given turn.

The second misuse is subtler. It's tempting to upload a document of example Q&A pairs or response templates and expect the agent to imitate them. The RAG guidance scopes knowledge to factual questions and answers, and explicitly excludes full document comparison and complex reasoning over long unstructured documents — which is exactly what pattern-imitation asks for.

The mechanics also fight you: retrieval returns only the top ~3 chunks per source, matched against the user's query. An exemplar isn't a factual answer to a user question, so it may never be retrieved — and if it is, you get fragments, not the whole template. The supported home for examples is few-shot prompting inside instructions, which Microsoft's declarative-agent guidance recommends for complex scenarios. Examples in instructions are always in context; examples in knowledge are maybe-retrieved fragments.

Gotcha

Nothing technically blocks you from uploading an "examples" document to knowledge — it just fails silently. Retrieval won't reliably surface it, and when it does, only fragments arrive. The agent seems to work in a demo, then ignores your template in production. Put examples in instructions.

The precise boundary

Instructions shape summarization — never retrieval

Here is the sharpest line in the docs, and the one that matters most for your agent's instruction set. After retrieval brings chunks back, your custom instructions govern the summarization step: tone, formatting, safety, brevity, flow. They apply after the facts come back (retrieval-augmented-generation, step 3). What they can never do is change which facts come back or how they're ranked.

Verified

"Agent instructions can't modify search retrieval logic. Remove any instructions that attempt to influence document retrieval." (generative-mode-guidance)

Gotcha

Instructions like "prefer document X over document Y" or "search only the policy folder" don't error — they just do nothing to retrieval, while adding noise to your prompt. If an instruction in your agent tries to steer what gets retrieved or how it's ranked, Microsoft's guidance is to delete it.

What to do instead

Reference sparingly, scope structurally

Two practical consequences for your agent's build:

  • Don't enumerate your sources in instructions. The orchestrator already routes using each source's name and description. The generative-mode guidance says to add hints only where the right tool or knowledge source is genuinely ambiguous — e.g. "for HR policy questions use Employee Handbook, not Vendor Contracts."
  • Scope retrieval with structure, not prose. By default generative orchestration searches every source on the Knowledge page. To restrict a scenario to specific sources, put them in a generative answers node inside a topic — node-level sources override the agent-level list, which then acts as a fallback.
Field note

The placement test for any content: does it tell the agent how to act, or does it tell the agent what is true? How-to-act (tone, routing, format, examples) goes in instructions. What-is-true (policies, product data, documentation) goes in knowledge. If it tries to steer retrieval, it goes nowhere — restructure instead.

How retrieval actually works

ScopeThe orchestration-driven RAG pipeline: who triggers search, the four steps, and the gates
Builds on01 · Knowledge vs. Instructions
Sourceresearch/copilot-studio-knowledge/03

The win: After this lesson you can trace any question a user asks your agent through the retrieval pipeline — predict whether knowledge fires, how many chunks come back, and where your configuration actually has leverage.

Who decides

The orchestrator triggers retrieval — per turn, automatically

You never wire "call knowledge here" on the default path. Under enhanced (generative) orchestration — the current default — the runtime evaluates each turn and decides whether grounding is needed and which sources to search.

Verified

"The enhanced orchestration runtime automatically determines which knowledge sources to search based on the user's question." (knowledge-copilot-studio, agents-experience)

This is why lesson 01's boundary matters: retrieval is the orchestrator's call, made from the user's question and your sources' names and descriptions — not from anything your instructions say.

The pipeline

Four steps, every grounded turn

StepWhat happensYour leverage
1 Query rewriting. The raw message becomes an optimized search query, enriched with conversation context (prior turns, location, time). None — automatic
2 Content retrieval. The rewritten query runs against all configured sources; Copilot Studio pulls the top three results from each source. Structural only — which sources exist and where
3 Summarization. Retrieved chunks become a grounded answer. Custom instructions apply here: tone, formatting, safety, brevity. Instructions bite here
4 Safety validation. Grounding, provenance, and relevance checks; the answer ships with citations. None — automatic
Verified

"After rewriting the query, the system runs it against all the knowledge sources you set up. Copilot Studio gets the top three results from each source." (retrieval-augmented-generation)

Gotcha

Top-3-per-source means adding sources widens coverage but never deepens it — each source contributes at most ~3 chunks no matter how relevant its content is. This is why RAG handles factual snippet questions well and whole-document reasoning badly, and why an agent can look well-grounded while missing most of a long document.

For public-website sources, retrieval runs through Bing Custom Search, restricted to your configured domains, with the same separation of retrieval from summarization plus grounding, provenance, and semantic-similarity cross-checks.

Indexing

How your files become retrievable

Uploaded unstructured files are chunked and vector-indexed (semantic embeddings) and stored in Dataverse. At query time, Copilot Studio retrieves the chunks most relevant to the rewritten query and grounds the response on them.

Directional

Chunk size is not configurable and not published by Microsoft (per practitioner reporting). Plan on it: you cannot tune chunking — your levers are document structure and source selection, not index parameters.

A doc nuance

Fallback vs. proactive search — both framings appear

You'll meet two descriptions of when knowledge search fires, and both are correct for their context:

  • Classic / SharePoint generative-answers wording: knowledge search runs as a fallback — it triggers when no topic answers the user, and for SharePoint it searches the configured URL plus all subpaths. (knowledge-add-sharepoint)
  • Modern generative orchestration (the default): the orchestrator invokes knowledge search proactively via the Universal Search Tool — not strictly as a last resort.
Field note

On generative orchestration, think "the orchestrator searches knowledge whenever it judges the turn needs it," not "knowledge only fires when topics miss." The fallback framing is incomplete for the modern default — not wrong for classic. Don't design your agent around last-resort retrieval.

The gates

Grounding enforcement and source scoping

Two settings decide what retrieval is allowed to do for your agent:

1. Allow ungrounded responses. This gate decides whether the agent may ever answer without grounding.

Verified

"When you turn off this setting, the agent blocks any response generated in a turn where it didn't use a knowledge source or tool… the fallback topic triggers." (knowledge-copilot-studio)

Off means every turn must ground on a knowledge source or tool, or the agent refuses and routes to the fallback topic — the setting for "answer only from approved content" agents. On means the agent may fall back to general model knowledge — more flexible, less contained.

2. Source scoping. By default, every source on the agent's Knowledge page is searched. To restrict a scenario to specific sources, use a generative answers node in a topic: node-level sources override the agent-level list, which then acts as a fallback. As lesson 01 established, this structural scoping is the only supported way to steer retrieval — instructions cannot.

Field note

Separate surface, don't conflate: the M365 Copilot Retrieval API is a developer API over the M365 hybrid index (KQL-scopable), distinct from Copilot Studio's point-and-click knowledge feature described here.

Knowledge source types & their limits

ScopePick the right source type for any content — and know the quota that will bite
Builds on02 · How retrieval works
Sourceresearch/copilot-studio-knowledge/04, 07

The win: given any piece of content for your agent — a policy library, a product site, a hand-written .md file — you can name the right source type in seconds, state its permission model and freshness, and know which limit (file size, input count, depth) will constrain the design.

The source map

Seven ways in — five that carry the weight

Copilot Studio's knowledge overview defines five core agent-level source types: public website (Bing Custom Search), uploaded files (chunked and vector-indexed in Dataverse), SharePoint (live GraphSearch), Dataverse tables, and enterprise connectors (Microsoft Search). Two more families round out the menu: OneDrive/SharePoint file-and-folder sync (specific files pulled into Dataverse, auto-synced) and third-party KB connectors (Confluence, Salesforce, ServiceNow, Zendesk — published articles only, collection-level). Azure OpenAI, Bing Custom Search, and Custom Data exist too, but only inside a generative-answers node's "Classic data" option — second-class under generative orchestration.

Two properties separate the sources more than anything else: who can see the content and how fresh it stays.

The numbers that shape a design

How many inputs a source type allows depends on orchestration mode — one more reason generative orchestration is the default for knowledge-heavy agents. Websites and SharePoint get 25 inputs in generative mode versus 4 in classic; Dataverse goes from 2 sources to unlimited. The agent-wide hard caps come from requirements-quotas:

Verified

An agent supports up to 500 knowledge sources and 500 uploaded files at up to 512 MB each; public websites index only two subpage levels below the configured URL (requirements-quotas, knowledge-add-public-website).

Field note

500 is a ceiling, not a target. Lesson 02's top-3-chunks-per-source rule means volume dilutes rather than deepens — curate a few well-named, well-described sources. Also: in generative mode, more than 25 sources get filtered by an internal model before retrieval; uploaded files are exempt from that count. And quotas shift often — re-verify against the live quotas page before you commit a design.

SharePoint is special

License gates and permission trimming

Two SharePoint behaviors dominate planning. First, the file-size ceiling for generative answers depends on licensing, per Microsoft's RAG guidance.

Verified

Without a same-tenant Microsoft 365 Copilot license, generative answers can only use SharePoint files under 7 MB. With the license plus tenant graph grounding and semantic search, the ceiling rises to 200 MB (retrieval-augmented-generation).

Second, SharePoint retrieval is identity-trimmed: per knowledge-add-sharepoint, calls run on behalf of the querying user, who must have at least Read permission. No access means the agent returns "no response" for that content. Good for governance — SharePoint knowledge can never broadcast restricted content.

Gotcha

Two silent failure modes. (1) An admin tester sees results a regular user won't — always test as a normal-permission user. (2) The SharePoint connector is blocked entirely if the tenant has Restricted SharePoint Search turned on, and guests in SSO apps are unsupported. Check both before betting a design on SharePoint.

File formats

Which path takes which files

The most confusing fact in this space: direct upload accepts the widest format set — Word, Excel, PowerPoint, PDF, plus .txt, .md, .html, .csv, .xml, .json, .yaml, EPUB, RTF, LaTeX and more — while OneDrive/SharePoint file-sync accepts only Word, PowerPoint, PDF, and Excel. So your hand-authored .md files cannot auto-sync; convert to docx/PDF if sync matters. Universal rules: 512 MB max; no image/video/audio files (images work only embedded in a PDF with alt-text).

Format quality in brief: clean prose (docx, PDF, .md) under clear headings chunks and retrieves best. Spreadsheets are the weak spot — RAG is poor at reasoning over tables, so push structured data to a SharePoint list or Dataverse table instead. PDFs added via SharePoint file-sync earn page-level citations — the answer links to the exact page.

Gotcha

Encrypted, password-protected, or sensitivity-labeled files index silently as "ready" but return nothing at query time. Same silent-blocker family: uploaded files require Dataverse search to be enabled in the environment — an admin action that fails quietly when missing.

The builder's playbook: what goes where

ScopePlace any piece of content — instructions, knowledge, or topic — correctly, first try
Builds onLessons 01–02 · 03 · Source types & limits
Sourceresearch/copilot-studio-knowledge/05

The win: the capstone of Course A. Hand yourself any piece of content for your agent — a persona, a policy PDF, a response template, a routing rule — and you can name where it belongs (instructions, knowledge, or a topic node), catch the six classic misplacements before they ship, and run the config checklist from memory.

The one rule

Facts get retrieved; behavior gets instructed

Everything in Course A compresses into one sentence: knowledge = facts the agent retrieves and cites; instructions = behavior, routing, tone, and any examples or templates. Two symmetric mistakes fall straight out of it. Putting behavior into knowledge fails because retrieval is conditional and fragmentary — top-3 chunks, only when the orchestrator searches. Putting retrieval-steering into instructions fails because instructions shape summarization, not retrieval.

Verified

Instructions drive the agent's behavior and cannot modify what retrieval returns; Microsoft's guidance is to remove instructions that try to steer retrieval and to scope sources structurally instead (authoring-instructions, generative-mode-guidance).

Anti-patterns

Six misplacements to catch in review

  1. Retrieval-steering instructions — "always prefer the 2025 policy doc," "search only the HR folder." Instructions can't do this; scope with a generative answers node.
  2. Examples or templates stored as knowledge. Retrieval is conditional and top-3-fragmentary — your exemplar may never surface. Put examples in instructions.
  3. Enumerating every source in instructions. The orchestrator already routes on source names and descriptions; boilerplate listings waste instruction space.
  4. Dumping hundreds of sources for "coverage." ~3 chunks per source per turn means volume dilutes. Curate instead.
  5. Expecting knowledge to enforce persona. Knowledge is data; it cannot define behavior.
  6. Planning large SharePoint docs without checking licensing. 7 MB without a same-tenant M365 Copilot license vs. 200 MB with it (RAG guidance).
Gotcha

Anti-patterns 1–3 fail silently: the agent still answers, just ignores your steering line or never retrieves your template. Nothing errors — quality quietly degrades. That's why they must be caught in review, not discovered in production.

Config decisions

Five choices to make deliberately

Make these explicit — never defaults-by-accident:

  • Orchestration → generative. More inputs, proactive automatic retrieval, node-level scoping — the recommended mode for knowledge-heavy agents (knowledge overview).
  • Allow ungrounded responses. Off = must ground every answer (high assurance; refuses and hits the fallback topic when it can't). On = may fall back to general model knowledge.
  • Source curation. Clear, distinct name + description per source — this is the routing signal. Few authoritative sources beat many.
  • Scoping. Topics that must draw only from certain sources get a generative answers node listing exactly those sources.
  • SharePoint verification. Confirm the license-gated size ceiling and test permission trimming as a regular user, not an admin (knowledge-add-sharepoint).
Field note

Need templated or example-driven responses? Knowledge is the wrong vehicle. The documented path is few-shot input→output pairs in instructions; topics + prompt nodes are the fallback for deterministic structure. Prototype the instructions approach first.

The checklist

Run this before you ship

  • Behavior, persona, tone, and examples live in instructions — not knowledge.
  • No instruction tries to steer which documents get retrieved.
  • Retrieval scoping (where needed) is done via generative answers nodes.
  • Every source has a clear, distinct name + description.
  • Sources are curated, not dumped (top-3-chunks-per-source).
  • Allow ungrounded responses is set deliberately for your assurance level.
  • SharePoint: licensing, size ceiling, and permission trimming confirmed.
  • Within the caps (500 sources / 500 files) — verified against live quotas, since numbers shift.
  • At large scale: evaluated the M365 Retrieval API / Azure AI Search path vs. built-in knowledge.

The two SharePoint RAG paths

ScopePath A (connector) vs. Path B (file upload) — and why every tuning decision depends on which one you chose
Builds on03 · Knowledge source types
Sourceresearch/sharepoint-rag-optimization/01

The win: Before touching any retrieval knob on your agent, you can name which of the two SharePoint RAG paths it uses — and therefore which levers (search schema and filters vs. file hygiene) actually apply, and why only the top three search results ever matter.

The core distinction

Two paths, not one

Copilot Studio grounds on SharePoint two architecturally different ways. They are not interchangeable, and mixing up their mechanics is the root of most wasted tuning effort.

Path A — SharePoint knowledge source (the connector). You point the agent at a SharePoint URL. At query time it queries SharePoint's live search infrastructure in real time via GraphSearch. Nothing is copied — content stays in SharePoint. It authenticates as the querying user's Entra ID, supports filters (title, author, modified by/date) and sensitivity labels, and allows up to 25 URLs under generative orchestration (4 in classic).

Path B — File upload. Files are copied into Dataverse, where they are chunked and vector-indexed into Copilot Studio's own semantic index. Retrieval matches query intent against those embeddings — and content is synchronized only every 4–6 hours, not in real time.

What Path A rides on

The tenant semantic index

Because your tenant is M365 Copilot licensed, Microsoft automatically builds a tenant-level semantic index from text-based SharePoint Online files. Each item becomes a numerical vector; semantically similar items cluster together, enabling similarity search "beyond exact keyword matching."

Verified

Field note

How content enters the index

Crawled properties → managed properties

SharePoint indexing runs on a two-tier property model. A crawl extracts content and metadata from each item as crawled properties. You then map crawled properties to managed properties in the search schema — and set attributes like Searchable, Queryable, and Retrievable. Only managed-property content reaches the index that GraphSearch (and therefore your agent) queries.

Verified

"To include the content and metadata of crawled properties in the search index, you map crawled properties to managed properties. The index only includes content and metadata from the managed properties." — crawled-and-managed-properties-overview, manage-search-schema

Two consequences: metadata that isn't mapped to a managed property is invisible to the agent, and a new mapping only takes effect after the content is re-crawled.

Directional

The crawled/managed model is documented on SharePoint Server pages. The attribute semantics carry over to SharePoint Online, but SPO manages the schema per-tenant with less crawl-scheduling control — treat attribute meanings as authoritative and crawl-timing mechanics as directional.

Inside the black box

Chunking and embedding — known vs. unknown

On Path B, files are chunked and embedded as vectors in Dataverse (knowledge-unstructured-data, verified); on Path A, content is vectorized into the tenant semantic index (verified, above). Past that, the pipeline is a black box.

Unconfirmed

Chunk size and chunking strategy are not documented and not configurable — and whether document structure (headings, sections) affects chunk boundaries is not confirmed by any primary source. Standard RAG intuition says it should; Microsoft doesn't say. Treat as a hypothesis to test, not a fact.

The consequential number

Only the top 3 results ground each answer

This is the single most consequential retrieval fact for tuning your agent.

Verified

"When Copilot Studio searches SharePoint, only the top three search results are used to summarize and generate a response. If no search results are returned, the generative answers node doesn't provide a response." — sharepoint-no-response

Gotcha

More documents ≠ better answers. If the right document isn't in the top 3, it doesn't ground the answer at all — and near-duplicate documents are actively harmful, crowding each other out of the top 3 and splitting relevance. Precision and disambiguation (clear titles, distinct content, filtered scope) beat volume, every time.

Permission trimming & security

ScopeHow SharePoint knowledge enforces permissions at query time — and why "no response" is usually a permissions problem
Builds on05 · The two SharePoint RAG paths

The win: When a user of your agent reports "it just says nothing," you diagnose permissions first — not content, not config — and you can explain exactly why the agent stayed silent. You also know to audit your site's permissions before rollout, because the agent will surface everything users can technically read.

The mechanism

Trimming happens at retrieval time, per user

Across every SharePoint path — connector, file upload, and the Retrieval API — permission trimming happens live, at query time, using the querying user's Microsoft Entra ID (delegated permissions). The system checks access against the source before returning or summarizing anything, and the user needs at least Read permission on the relevant sites and files (knowledge-add-sharepoint).

This holds even for Path B, where content was copied into Dataverse: no permission information is stored locally — every check goes back to SharePoint.

Verified

"Even though the system stores chunks and indexes locally in Dataverse, it performs a live check on the queries to make sure the current user has access to the data before providing a summary or response… The system doesn't store content permission information locally… all permission checks live with the source." — knowledge-unstructured-data

Field note

Governance upside: you cannot use a SharePoint knowledge source to broadcast restricted content to users who lack access. The trimming is identity-based and enforced at the source, every query — the same model the Retrieval API documents ("individuals can only get results from the content they are allowed to access").

The #1 troubleshooting trap

The silent "no response"

When the user lacks access, the agent does not error. It returns nothing — silently.

Verified

"Generative answers over SharePoint rely on delegated permissions… a user must have read permissions on the relevant sites and files, or no search results will be returned… nor any errors or exceptions." — sharepoint-no-response

Gotcha

There is no error, no exception, no "access denied" — just "no response." Nothing in the transcript tells you the cause was permissions. If you treat this as a content or configuration bug, you will burn hours tuning documents that were never the problem.

Now combine this with lesson 05's top-3 rule: if the three top-ranked results are documents the user can't see, they are trimmed away and nothing remains to ground on — the user gets no answer even though relevant, accessible content exists at rank 4 and below.

Verified

Inaccessible top hits can leave a user with nothing: permission trimming (verified) plus top-3 grounding (verified) means a user who can only see documents ranked 4th or lower gets no answer at all. This is an inference from two verified facts, flagged as such in the research (sharepoint-no-response).

Testing implication

Test as a real end user — never as yourself

Trimming is per-identity, and you are the site owner. An admin or site-owner tester sees results a regular user won't. Always test with an account that has the actual permissions of your target audience (knowledge-add-sharepoint).

Directional

If answers work for you but users report "no response," suspect permissions first — before content or configuration. This triage order comes from the verified mechanics but isn't itself a documented procedure.

Field note

For your agent: keep (or borrow) a test account with exactly the target audience's permissions, and make a permission-trimmed test pass part of every rollout. Your own account is the least representative one in the tenant.

The flip side

Oversharing — trimming is only as good as your permissions

Permission trimming enforces what users can access, not what they should access. If a library is over-permissioned (say, "Everyone" has Read), the agent will happily ground answers on it for everyone who asks.

Directional

Copilot agents surface content users technically can access but perhaps shouldn't — latent oversharing becomes suddenly very visible and very queryable. Site owners should audit and tighten permissions before rollout. (Practitioner-sourced; Microsoft's privacy doc confirms the trimming model, but the remediation guidance is community wisdom.)

One more enforcement layer, verified for the connector path: it respects sensitivity labels in addition to Read permissions (knowledge-unstructured-data).

The stakes

Top-3 grounding makes precision decisive

Directional

The pipeline

Crawled → managed: nothing is searchable until it's mapped

SharePoint search discovers metadata as crawled properties, but only managed properties participate in queries and results (crawled-and-managed-properties-overview). New metadata joins the index only after you map it and the content is re-crawled.

Verified

"If you add a managed property, you must map it to a crawled property to get content into the index. Field note

Translation: it's a query door straight into the tenant semantic index your M365 Copilot license already maintains. No sync cadence, no copied files, no second index to secure. But it's an API — you write the code that calls it — and it's currently in Preview.

The constraint

Format limits — the sharpest fact in this lesson

The API's documentation is unusually explicit about what it can and cannot retrieve, and these limits are the main thing to memorize.

The decision

Which grounding path, when

Use…When

Directional

For most site-owner-driven Copilot Studio scenarios — including yours — the built-in connector is the right default, and the search-schema/filters playbook (lesson 08) is where your effort pays off. Reach for the Retrieval API only when you're building a custom grounding pipeline and need KQL-level control.

The common thread

All three paths ride your tenant semantic index

Whether grounding flows through the connector, file upload, or the Retrieval API, licensed retrieval rides the same automatically built tenant semantic index.

Field note

This is why the document-hygiene work from lesson 09 and the permission work are never wasted: improving your SharePoint content and permissions improves every grounding surface at once. Pick the path per scenario; the content investment transfers.

Case study: your own agent's build review

The setup

Nine findings, zero new concepts

On 2026-07-21 a technical review read your shipped Agent/ package — settings, instructions, README, and the five .md knowledge files — and checked its risk claims against live Microsoft Learn docs. It found nine issues, none requiring a redesign. The point of this lesson is not the fixes (they're one-line edits). It's that almost every finding is an instance of a distinction this course already taught. If the concepts stuck, you should be able to predict each finding before you read it.

Verified

One worry the review cleared up front: Copilot Studio's supported knowledge file types explicitly include "Text (.txt, .md, .log)" — so uploading the five Markdown knowledge files as-is works. (Upload files as a knowledge source.)

The review groups the findings worst-first: things that stop the build, things that degrade the running agent, and things that rot quietly over time. We'll walk them in that order.

Group 1 · Build-blocking and setup gaps

The build stops before it starts

Finding 1 — the Dataverse search prerequisite never reached the build guide

What: Agent/README.md lists only a Copilot Studio license and "permission to add knowledge sources" as prerequisites. Why it matters: you learned in the fundamentals course that the file-upload path copies files into Dataverse and indexes them there — which is exactly why the environment must have Dataverse search switched on before any file can be added.

Verified

Uploading files as an agent-level knowledge source requires Dataverse search enabled in the environment — an admin setting. If it's off, you cannot add file knowledge at all. (Upload files as a knowledge source.)

Gotcha

The agent's own knowledge already documents this requirement (knowledge/03-artifact-formats.md:377) — it just never made it into the setup doc a human follows. Knowing a fact and operationalizing it are different ship gates. Audit your build guide against your own knowledge files.

Fix

Add "Dataverse search enabled in the target environment (admin action)" to the README prerequisites. One line, saves the most likely step-4 dead stop.

Finding 2 — "add these three" starters, followed by four rows

00-agent-settings.md says "Add these three" conversation starters and then lists four (Build / Fix / From my docs / Not sure); the README repeats "three." Trivial to fix, but it's the maintenance lesson in miniature: any hard-coded count is a claim that can drift from the thing it counts. Pick one — drop a row or say four.

Finding 3 — five knowledge sources, zero descriptions

What: the README says "upload all five files" and never mentions writing a Description for each. Why it matters: this is description-driven source routing from the fundamentals course. Under generative orchestration the agent decides which source to consult by its name and description. Your own knowledge/04-platforms-and-quality.md calls description quality "the main authoring lever" — and the Learn upload flow makes Description an explicit step.

Gotcha

Five undescribed sources don't error. The agent still answers — it just routes unreliably, pulling from the wrong file with full confidence. Routing failures are silent; you only see them as "the answer felt off."

Fix

Write a one-line description per file into 00-agent-settings.md and add "enter each description" to the upload step. Descriptions are the cheapest retrieval-quality lever you have.

Group 2 · Runtime reliability

The agent runs — differently than you think

Finding 4 — "Allow ungrounded responses = ON" fights the knowledge base

What: the instructions demand "Ground every fact… never invent platform behaviour, schema fields," yet the ungrounded-responses toggle is ON. With ungrounded ON, nothing enforces retrieval: the model can answer schema versions, limits, and portal steps from its own — possibly stale — parametric memory instead of the five files.

Why it happens: this is the ungrounded-responses trade-off from the fundamentals course, not an oversight. With the toggle OFF, clarifying and follow-up questions are treated as ungrounded and stop working. The setting was turned ON to keep the conversation usable — and the hallucination risk came along for the ride.

Fix

Keep it ON, but harden # SCOPE & GROUNDING: "before stating any schema field, version number, or limit, retrieve it from knowledge; if not retrievable, say so." Then lean on Finding 5, so the agent's core method never depends on retrieval in the first place.

Finding 5 — the method lives in knowledge; it belongs in instructions

What: the agent's operating procedure — the 5-layer isolation algorithm, error-code decoder, workflow routing — lives in the knowledge files. Why it matters: this is the course's core distinction, applied. Instructions drive behavior and are always in the model's context. Knowledge is best-effort RAG: Copilot Studio chunks the documents, and semantic retrieval can return a partial chunk — half a table, three steps of five — instead of the whole procedure. A method that must run completely should never be fetched probabilistically.

Verified

The famous 8,000-character instruction ceiling applies to declarative agents for Microsoft 365, not Copilot Studio custom agents. 01-instructions.md stays short and delegates the method to knowledge for no reason — there's room to promote it.

Fix

Promote the non-negotiable procedure — greet/route logic, Workflow A/B steps, the five-layer order — into the instruction body. Leave knowledge to what it's good at: lookup detail (schemas, error codes, archetypes). Instructions = guaranteed context; knowledge = best-effort facts. This is also the best hedge against Finding 4: a method carried in instructions can't fail to be retrieved.

Finding 6 — moderation "High" can filter the agent's own answers

What: this agent talks about jailbreaks, prompt injection, and XPIA constantly — it's a security-aware builder's assistant, and users paste injection-looking artifacts at it for diagnosis. Why it matters: this is moderation self-filtering from the diagnostics material: High RAI moderation can trip OpenAIJailBreak / OpenAIndirectAttack on exactly the content this agent must discuss. Your own knowledge/05-diagnostics.md documents it as a failure mode.

Gotcha

The filter doesn't distinguish discussing an attack from performing one. An agent whose subject matter is agent security will periodically censor itself — and the user just sees a blocked or deflected answer, not the reason.

Fix

Consider Medium, or ship High expecting to notch down. Either way, add the documented mitigation to instructions: a line stating that discussing agent security is expected behavior.

Finding 7 — five sources, sitting exactly at the ~5-source ceiling

What: the package ships five knowledge sources. Why it matters: per your own research (knowledge/03-artifact-formats.md:375), only about five sources are effectively usable at a time under generative orchestration, which filters beyond that. At exactly five you have zero headroom and maximum routing contention — every new source you add later degrades routing for all of them.

Fix

Consolidate into fewer, larger, well-described sources — e.g. merge 01+02 into a single build reference and keep 05 diagnostics separate — or use file groups. Fewer, sharper sources beat many overlapping ones (the same curation logic as SharePoint's top-3 grounding).

Group 3 · Accuracy and maintenance

Correct today, wrong at the next version bump

Finding 8 — hard-coded versions drift, and ungrounded ON makes the drift silent

What: the knowledge pins declarativeAgent v1.7, apiPlugin v2.4, app manifest 1.19, a Copilot-Credit rate table, and GPT 5.0→5.1. Why it matters: this is version drift compounded by Finding 4. When Microsoft bumps a schema, the agent keeps emitting the stale version confidently — and because unrecognized properties invalidate an entire manifest, one field blended from parametric memory produces a document that fails validation. The package ships zero tools by design, so nothing in the loop catches it.

Gotcha

A concrete landmine is already in the files: the archetype YAML uses lowercase web_search / code_interpreter (knowledge/02-archetype-library.md:65,165), but real manifest capability names are WebSearch / CodeInterpreter. Echo the archetype casing into JSON output and the manifest is invalid — with no error until the user tries to deploy it.

Fix

Add a "verify current schema version before emitting" self-check to instructions, and make the AgentSpec→manifest casing rule explicit in 03-artifact-formats.md. Longer term, a validation flow (already noted as a future enhancement) closes the loop properly.

Finding 9 — (minor) the Fallback topic isn't your out-of-scope guardrail

The README says to edit the classic Fallback system topic. Harmless — but under generative orchestration, off-topic handling is driven mostly by the # SCOPE / # GUARDRAILS instructions, not that topic. Keep the instruction-level refusal as the real control and treat the topic edit as cosmetic. Same lesson as Finding 5: behavior belongs in instructions.
